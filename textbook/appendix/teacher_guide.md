# 教師用ガイド

## 教科書の設計思想

本書は「ロボット犬」という身近で具体的な題材を通じて、
Java とオブジェクト指向プログラミング（OOP）を学ぶ教科書です。

### 3 つの学習レイヤー

```
レイヤー3：ロボティクス設計（世界モデル・ROS・マルチエージェント）
レイヤー2：OOP 設計（継承・インターフェース・多態性・戦略パターン）
レイヤー1：Java 文法（クラス・変数・メソッド・コンストラクタ）
```

各課はこの3層を同時に学べるように設計されています。

## 各課の指導ポイント

### 第0課：Java にようこそ
- **最重要ポイント**: プレイグラウンドで「動いた！」という体験をさせる
- **注意点**: 環境構築でつまずくとやる気が下がる。最初はブラウザ上のプレイグラウンドを使うのが安全
- **発展課題**: System.out.println で ASCII アートのロボット犬を描く

### 第1課：クラスとオブジェクト
- **最重要ポイント**: 「設計図」と「実物」の違いを物理的な例えで理解させる
- **注意点**: `new` キーワードの意味が抽象的なので、繰り返し説明する
- **アクティビティ**: 教室で「人間クラス」から「生徒オブジェクト」を作るロールプレイ

### 第2課：カプセル化
- **最重要ポイント**: 「なぜ隠す必要があるのか」を実感させる
- **注意点**: getter/setter の「おまじない感」を減らすために、ないと何が困るかを先に体験させる
- **アクティビティ**: 悪意のあるコードでバッテリーを -999 にする「攻撃」をやってみせる

### 第3課：コンストラクタ
- **最重要ポイント**: 「忘れずに初期化する」ための仕組み
- **発展課題**: コンストラクタ内でファイル読み込みやネットワーク接続を行う例

### 第4課：継承
- **最重要ポイント**: 「is-a」関係の理解
- **アクティビティ**: クラスの階層構造をホワイトボードに書かせる
- **落とし穴**: 継承よりコンポジションの話はここでは深入りしない（第7課で扱う）

### 第5課：インターフェース
- **最重要ポイント**: 「約束事」としてのインターフェース
- **注意点**: 継承との違いが曖昧になりがちなので、表で比較する
- **アクティビティ**: 犬以外の動物（Cat, Bird）でも Dog インターフェースを実装させてみる

### 第6課：多態性
- **最重要ポイント**: この課が OOP のクライマックス
- **アクティビティ**: if-else で書いたコードを多態性で書き換えるリファクタリング演習
- **発展課題**: `instanceof` を使うべきでない理由を議論する

### 第7課：戦略パターン
- **最重要ポイント**: 「継承よりコンポジション」の実践
- **アクティビティ**: グループごとに異なる頭脳を実装して、同じエージェントで動かす
- **発展課題**: デコレーターパターン（LoggingBrain）への発展

### 第8課：AI エージェント
- **最重要ポイント**: 状態・行動・頭脳の3要素
- **アクティビティ**: ロボット掃除機（Roomba）を例にしたエージェント設計
- **発展課題**: 強化学習とのつながりを紹介

### 第9課：世界モデル
- **最重要ポイント**: 「予測」という思考の導入
- **注意点**: ここで挫折する生徒が出ないよう、シンプルな評価関数から始める
- **発展課題**: 複数ステップの先読み、MinMax アルゴリズムへの発展

### 第10課：pub/sub
- **最重要ポイント**: 疎結合の価値
- **アクティビティ**: 実際の ROS の画面を見せる（rviz など）
- **発展課題**: MQTT などの他のメッセージングシステムとの比較

### 第11課：マルチエージェント
- **最重要ポイント**: 「協調」と「役割分担」
- **アクティビティ**: クラス全体で一つのシミュレーションを動かす
- **発展課題**: 競合（リソースの取り合い）の実装

## 自由課題のアイデア

1. **ロボット犬シミュレータ**
   複数の頭脳・複数の犬・複数のトピックを組み合わせた完全なシミュレータ

2. **迷路探索ロボット**
   壁センサーと世界モデルを使って迷路を探索するロボット

3. **犬のしつけシミュレータ**
   報酬と罰を使って犬の行動を学習させる（Q-learning の導入）

4. **災害救助チーム**
   複数のロボット犬が協力して被災者を探すシミュレーション

## 評価のポイント

| 評価項目 | 割合 | 評価基準 |
|---------|------|---------|
| 演習問題 | 40% | 各課末の問題。コードが正しく動作すること |
| 総合演習 | 30% | 自由課題。完成度 + 工夫 + コードの品質 |
| 発表 | 30% | プレゼンテーション + 質疑応答 |

### コード品質の評価基準

- カプセル化が適切に使われているか
- 継承・インターフェースが適切に使われているか
- 多態性を活用できているか
- 無駄な重複コードがないか
- わかりやすい名前が使われているか

## トラブルシューティング

### よくあるつまずき

| 症状 | 原因 | 対処 |
|------|------|------|
| `static` がわからない | 静的メソッドとインスタンスメソッドの混同 | 「クラスに属する vs オブジェクトに属する」の図解 |
| `this` の意味がわからない | 自分自身の参照概念が難しい | 「クラスの中から自分を見る」という比喩 |
| 継承とインターフェースの違いがわからない | 抽象概念のオーバーロード | 表で比較、具体例を増やす |
| コンパイルエラーが直せない | エラーメッセージが読めない | 代表的なエラーと対処法の一覧表を配布 |
