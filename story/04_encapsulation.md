そこで見たのは——保護カバーと、触ってはいけない基板。

「外から触っていい場所と、触っちゃダメな場所があるんだ…」

## 本文

「シロ、お手」

シロはゆっくりと前脚を上げた。

「こりゃあ、ハードウェアから直さないと」

タクミはドライバーセットを取り出した。
シロを机の上に寝かせて、背中のネジを外し始める。

カバーを開けると、中にはぎっしりと基板と配線が詰まっていた。

「すげえ……本格的だ……」

でも、どうやって直せばいいんだろう？
タクミはサービスマニュアルを開いた。

---

## カプセル化（ハードウェア版）

サービスマニュアルにはこう書いてあった。

```
【警告】
- 基盤A-3（バッテリー制御モジュール）は保護カバーで覆われています
- このカバーを外すと保証対象外になります
- 基板A-3に直接アクセスする必要は通常ありません
- 充電は外側のUSB-Cポートから行ってください
```

つまり——
- 充電するだけなら、外側のポートにケーブルを差せばOK
- 基板A-3の中身を知らなくても、ロボット犬は使える
- 保護カバーは「触ってはいけない場所」の印

「これ、プログラムも同じじゃないか？」

タクミは昨日書いたコードを見直した。

```java
class RobotDog {
    String name;
    int battery;
}
```

「今、`battery` ってプログラムのどこからでも触れるようになってる。
でも、バッテリーの管理って、外から適当に触られたら困るはず…」

Java には「アクセス修飾子」というものがある。
`private` をつけると、そのクラスの外からは触れなくなる。

```java
class RobotDog {
    private String name;
    private int battery;

    void charge() {
        battery = 100;
        System.out.println("充電したよ！");
    }

    int getBattery() {
        return battery;
    }
}
```

「これで、`battery` は外から直接触れなくなった。
バッテリーを操作したかったら `charge()` っていうメソッドを通すしかない」

それが——**カプセル化**。

タクミは笑った。シロは首をかしげた。

---

## 知ったこと

- **private**：自分のクラスの中からしか触れない
- **カプセル化**：データを隠して、必要な操作だけ公開すること
  - ハードウェアでは「保護カバー」
  - ソフトウェアでは「private」
- **getter**：値を取得するためのメソッド（`getBattery()`）
- **メリット**：中身を壊されない、後で中身を変えても外に影響しない
