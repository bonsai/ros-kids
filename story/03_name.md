# 第3話　名前をつけた

## あらすじ

毎回プログラムを書き換えて名前を変えるのに疲れたタクミは、
「名前を外から変えられる仕組み」を調べる。

そこで出会ったのが——**変数**と**クラス**の本当の意味。

---

## 本文

「毎回 `Main.java` を書き換えるの、めんどくさい…」

タクミは3回目の名前変更でそう思った。

最初は「RS-01」。
次に「シロ」。
そして「クロ」にしようとして——プログラムを直すのが面倒になった。

タクミは検索した。

「プログラム　値を変える　直さずに」

出てきたのは「変数（へんすう）」という言葉。

```java
String name = "シロ";
```

「String name」……文字列を入れる箱？
「= "シロ"」で中身を入れている？
これなら、この一行だけ書き換えれば名前が変わる。

```java
public class Main {
    public static void main(String[] args) {
        String name = "シロ";
        greet(name);
    }

    static void greet(String dogName) {
        System.out.println("こんにちは");
        System.out.println("ぼくは" + dogName + "です");
    }
}
```

コンパイルして転送。

```
---
こんにちは
ぼくはシロです
---
```

「できた……！ これで名前を変えるときは `String name = "シロ";` のとこだけ直せばいい」

でも。タクミは気づいた。

「これ、Main っていうクラスが『プログラム全体』になってて、
ロボット犬自体がどこにもいない……。

ロボット犬を——『モノ』としてプログラムで表現できないかな？」

---

検索してたどり着いたのは、「クラスとオブジェクト」という考え方。

クラスは「設計図」。
オブジェクトは「設計図から作った実物」。

```java
// 設計図
class RobotDog {
    String name;
}
```

```java
// 実物を作る
RobotDog myDog = new RobotDog();
myDog.name = "シロ";
```

タクミはひらめいた。

「ロボット犬を『クラス』として設計すれば、
いくらでも実物を作ったり、改造したりできるんだ！」

タクミはコードを全部書き直した。

```java
class RobotDog {
    String name;

    void greet() {
        System.out.println("こんにちは");
        System.out.println("ぼくは" + name + "です");
    }
}

public class Main {
    public static void main(String[] args) {
        RobotDog myDog = new RobotDog();
        myDog.name = "シロ";
        myDog.greet();
    }
}
```

コンパイル。転送。起動。

```
---
こんにちは
ぼくはシロです
---
```

タクミは画面を見て、にやけた。

「これで……シロが『プログラムの中』に生きてるみたいになった」

シロ（ロボット犬本体）は、首をかしげたままタクミを見ている。

「お前、今プログラムの中で『オブジェクト』になってるんだぞ」

シロは——わかってないような顔をした。

## 知ったこと

- **変数**：値を入れる箱。`String name = "シロ";`
- **クラス**：設計図。`class RobotDog { ... }`
- **オブジェクト（インスタンス）**：設計図から作った実物
- **new**：実物を作る命令。`new RobotDog()`
- **プロパティ**：オブジェクトが持つデータ。`name`
- **メソッド**：オブジェクトが持つ動き。`greet()`

## 次回予告

動くようになったシロだけど、触ってみるとガタガタだ。
「中身、めっちゃ壊れてる……！」

タクミはシロのハードウェアを分解し始める。
そこで見たのは——「隠された設計図」と「触ってはいけない場所」。
